## Log (newest first)

### 2026-07-05

Sync bug — "one file per attempt", root-caused to a missing `return`. Symptom: first sync moved 2 of 6 notes, each later attempt one more, some never landing. Cause ([daemon/index.html](daemon/index.html)): `pushNote` kicked off its `fetch` without returning the promise, so `reconcileAll`'s reduce chain didn't wait — all pushes fired concurrently. GitHub's Contents API creates one commit per PUT parented on the branch HEAD, so concurrent writes against the same parent let only one commit win per round; the rest 409'd, and a 409 on a not-yet-existing file made `handleClash` fetch a 404 and no-op. Next run one more won → the exact incremental pattern. Fix: `pushNote` now `return`s its chain (and a resolved promise from the guard), serialising the PUTs so each commit parents cleanly on the last. Durable lesson: any async primitive a sequencer drives must return its promise — a silent missing `return` degrades "sequential" into "concurrent", and for GitHub's single-branch commit model that surfaces as one-write-per-round, not an obvious error.

Auto-sync fix — load `syncOn`/`syncRepo` at startup, not just when the ⚙ panel opens. Symptom: a note connected fine but read "never synced" and nothing reached GitHub. Root cause: the two flags were only populated in `loadSettingsPanel()` (on panel open), so at page load `connect()`'s `ws.onopen` and the 3-min poll both saw the default `syncOn=false`, `syncReady()` was false, and every auto-trigger silently skipped. Fix ([daemon/index.html](daemon/index.html)): new `loadSyncConfig()` fetches `/api/settings` right after auth succeeds, sets the globals, and kicks an immediate `reconcileAll('startup')` when ready. Wired into all three post-auth entry points — `checkAuthAndInit` success, the PIN-poll success, and the manual PIN submit — so sync fires whether the page loads already-authed or the user logs in via PIN. Durable lesson: an auto-trigger that reads config is only as good as the earliest point that config is loaded — load enablement flags at init, never lazily on a settings-panel open.

Phase 9 — PIN hardened: 6 digits + per-IP brute-force lockout (device-verified). Bumped the auth PIN from 4 to 6 digits (1,000,000 combinations) and added a throttle: 5 consecutive wrong guesses from one IP locks *that IP* out for 60 s (HTTP 429 + `Retry-After`); the counter resets on success or when the lockout expires. Per-IP, not global, on purpose — a malicious LAN actor hammering guesses can't lock out the owner, who connects from a different address. State is in-memory (`map[ip]*pinAttempt` under a mutex, expired entries pruned each attempt), so a reboot clears it — fine, since a reboot also regenerates the PIN. Math: 5 tries/60 s vs 1 M combos ≈ 700 h to cover half the space — brute force is dead. Also switched the compare to `subtle.ConstantTimeCompare` (no timing side-channel; the lockout is the real defense). Bug caught while writing: `int(Uint32(...)) % 1000000` overflows the device's 32-bit `int` — do the modulo in `uint32` space first (`Uint32(b)%1000000`) so the value always fits. No keywriter rebuild — the Lobby interpolates the runtime PIN, so 6 digits render with an rmkbd-only deploy. Client: `maxlength=6` + a 429 "wait a minute" message.

Phase 9 — upload a `.md` file from the phone (device-verified). The second half of Upload (paste-text half already shipped). - Security review (Opus) — upload adds no new injection or traversal surface. Path traversal is impossible server-side: `notesSafe()` rejects `/` and `..` and force-appends `.md`, so a bypassed client can't escape the notes dir or write a non-`.md` path. Overwrite is impossible: the POST `os.Stat`s first and 409s rather than clobbering. Stored-XSS is inert: a malicious `.md` is read back via `textContent` (never `innerHTML`), and served as `text/plain` (read) / `text/markdown` + `attachment` (download), so the browser never executes it.

Phase 9 — auth remembered until the next 04:00 (shipped). The PIN cookie (`rmkbd_token`) now expires at the next local 04:00 instead of on session end, so a full day's writing never re-prompts while a fresh morning asks once. `nextMorningCutoff(now)` returns today's 04:00 if we're before it, else tomorrow's; `pinHandler` sets both `MaxAge` (modern browsers win) and `Expires` (older-browser fallback) at that moment. 8c cold-boot bug — `203/EXEC` race on the late-mounting `/home` partition (fixed, re-verified). First reboot booted into the stock GUI, no `:8000`. Boot journal: `Started … → Main process exited, code=exited, status=203/EXEC → Failed`, then 80 s later a clean manual `systemctl start`. Root cause: `203/EXEC` = systemd couldn't *execute* `ExecStart`. Our binary lives on `/home/root`, a separate partition that mounts later than rootfs; the unit reached `ExecStart=/home/root/rmkbd` before the mount, so the exec failed. Phase 7 accidentally dodged this because its `ExecStart` was `/usr/bin/systemd-inhibit` (on rootfs, always ready), which then launched `/home/root/rmkbd` a beat later — *after* the mount. Moving the inhibit wrapper into `launch-keywriter.sh` in 8c pointed `ExecStart` straight at the late-mounted binary and re-exposed the race. Fix: `RequiresMountsFor=/home/root` in the unit's `[Unit]` section. Re-verified: next reboot booted straight into the editor, single clean start, no `203/EXEC`. Durable lesson: any systemd unit whose `ExecStart`, working dir, or `EnvironmentFile` lives on `/home/root` MUST declare `RequiresMountsFor=/home/root` — rootfs-only paths mount early, `/home` does not.

Deploy transport — scp deadlocks on this link; gzip-over-ssh is the fix. `scripts/deploy.sh` wedged at a fixed ~255 KB every time (same byte offset, GUI up or asleep — so not sleep). scp's SFTP app-level windowing deadlocks on this Mac→Wi-Fi→tablet path; `IPQoS=throughput` + `ServerAlive` keepalives did not clear it. Fix ([scripts/_env.sh](scripts/_env.sh) `rm_send_file`): stream the file gzip'd through the SSH channel itself (`gzip -c src | ssh … 'gzip -dc > dst'`) — no SFTP windowing, and ~half the bytes — plus a post-copy `wc -c` size check to catch a truncated stream. Now the primary path for every deploy (binary + keywriter + Qt sysroot). Durable lesson: when scp stalls at a fixed offset on an embedded link, don't tune scp — switch transports to a raw gzip-over-ssh stream.

Boot auto-launches the editor (power-on = typewriter); Home restarts xochitl but rmkbd stays serving, so the phone re-launches writing with no reboot. ## Known prerequisites / gaps
- `/dev/uinput` is unavailable AND unfixable on kernel `5.4.70-v1.6.3-rm10x` (gate 🔴 RED; built but won't load — exports trimmed). The editor is fed over a local socket; no uinput.
- Go toolchain installed on the Mac (`brew install go`) — the only device-reachable host, so it cross-builds + deploys rmkbd (~1 s). CI/Docker stays for keywriter (Qt sysroot) only.
- Two-machine split: the work laptop can't reach the device (corporate VPN); device work runs on a second machine and syncs via git.
- Disk: only the tiny rootfs is tight, and nothing we ship goes there. `/` (rootfs, ~228 MB) is 96% full — but everything we deploy (keywriter binary + Qt5 sysroot ~14 MB + notes) lives on `/home/root/`, a separate multi-GB partition. Don't resize rootfs (A/B OTA scheme; brick risk).
